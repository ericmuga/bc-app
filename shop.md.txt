# Plan: Add Point of Sale Module to Existing Node.js + Vue App

## Goal

Add a Point of Sale (POS) module to the existing Node.js and Vue application, integrated with Microsoft Dynamics 365 Business Central through both API-based integration and database-to-database sync.

The POS must support offline operation at shop level, periodic synchronization to Business Central, stock control, payment reconciliation, MPESA payments, tax invoice signing, expense approvals, stock transfer requests, and role-based access control.

The Business Central repository will be shared and should be used as the main source of truth for business rules. The **SHOP Role Centre** in Business Central should be treated as the primary functional reference.

---

## Key Business Requirements

### 1. Business Central Integration

The POS module must support two integration modes:

1. **API integration**

   * Used for posting transactions, reading approved records, validating business rules, and calling Business Central codeunits/pages/API endpoints.
   * Should support authenticated requests to Business Central.

2. **Database-to-database integration**

   * Used for syncing master data and transactional records where direct database integration is required.
   * Must be implemented carefully with staging tables, sync logs, retry handling, and clear ownership of records.

Claude Code should inspect the shared Business Central repo and identify:

* Tables used by the SHOP Role Centre.
* Pages/actions exposed to shop users.
* Codeunits used for posting invoices, payments, stock transfers, tax signing, MPESA handling, and approvals.
* Custom table extensions, especially item and customer extensions.
* Existing AL integration logic that should be mirrored or called from Node.js.

## 2. Offline-First POS Operation

The POS must be able to work offline at shop level.

### Offline Capabilities

Shop attendants should be able to:

* View synced items.
* View synced prices.
* View synced shop customers.
* Create sales invoices.
* Receive payments.
* Record MPESA payment references where applicable.
* Perform daily stock/cycle counts.
* Request stock transfers.
* Request expense approvals.

### Offline Storage

Use a local browser database for the Vue app, preferably:

* IndexedDB via Dexie.js, or
* another reliable offline storage wrapper.

Data created offline should be stored locally with sync metadata.

Recommended sync fields:

```text
local_id
server_id
bc_document_no
sync_status: pending | syncing | synced | failed | conflict
sync_attempts
last_sync_attempt_at
last_sync_error
created_offline
created_by
shop_code
salesperson_code
created_at
updated_at
```

### Sync Job

A background sync job should run from the Node.js backend.

The job should:

* Push pending offline sales to Business Central.
* Push payments to Business Central.
* Push cycle counts and stock variances.
* Push stock transfer requests.
* Push expense approval requests.
* Pull updated master data from Business Central.
* Retry failed records.
* Log every sync attempt.
* Avoid duplicate posting in Business Central.

Recommended tools:

* BullMQ + Redis, or
* node-cron for simpler scheduling.

BullMQ is preferred if the app already has Redis or needs durable queues.

## 12. Roles and Permissions

The POS module must have at least two roles:

1. **Admin**
2. **Shop Attendant**

### Admin Role

Admin users can:

* Manage POS users.
* Assign users to shops.
* View all shops.
* View and manage synced master data.
* Trigger manual sync.
* View sync logs.
* Retry failed syncs.
* Approve or review expense requests, if configured.
* Review stock variances.
* View all reports.
* Configure payment methods.
* Configure integration settings.

### Shop Attendant Role

Shop attendants can:

* Access only their assigned shop.
* Create POS sales.
* Receive payments.
* Record MPESA references or initiate MPESA flow, depending on configuration.
* Perform daily stock counts.
* Add comments on stock variances.
* Request stock transfers.
* Request expense approvals.
* View their own shop reports.

Shop attendants must **not** be allowed to:

* Add master records.
* Edit master records.
* Delete master records.
* Change item prices.
* Create customers directly.
* Create items directly.
* Create salespersons directly.
* Change Business Central integration settings.
* Access other shops unless explicitly allowed.

### Permission Enforcement

Permissions must be enforced in both:

* Vue frontend route/UI guards.
* Node.js backend middleware/policies.

Do not rely only on hiding buttons in Vue.

Recommended backend middleware:

```text
requireAuth
requireRole('admin')
requireRole('shop_attendant')
requireShopAccess(shopCode)
```

## 16. Sync Design

Create a generic sync log table.

```text
sync_logs
```

Fields:

```text
id
entity_type
entity_id
direction: pull | push
source: pos | business_central
target: pos | business_central
status: pending | success | failed | conflict
attempt_count
request_payload
response_payload
error_message
started_at
completed_at
created_at
updated_at
```

### Sync Principles

* Business Central is the source of truth for posted financial and inventory transactions.
* POS can create transactions offline, but they must be synced and confirmed by Business Central.
* Master data should generally be read-only in POS.
* Shop attendants cannot create or modify master data.
* Avoid duplicate Business Central documents by using idempotency keys.
* Every POS transaction should have a unique local ID and external document number.
* Sync failures should be visible to admins.

## 19. Important Rules for Claude Code

When implementing this POS module:

1. Do not allow shop attendants to create, edit, or delete master records.
2. Do not duplicate Business Central business logic blindly; inspect the AL files first.
3. Use the SHOP Role Centre as the functional reference.
4. Prefer Business Central as the source of truth for posted financial and inventory records.
5. Make all offline-created records idempotent before syncing.
6. Add sync logs for every push and pull operation.
7. Make failed sync records visible and retryable by admins.
8. Enforce role and shop restrictions on the backend, not only in Vue.
9. Keep master data sync separate from transaction sync.
10. Keep tax signing and MPESA behavior aligned with the provided Business Central AL files.
